# Projection Engine — NUM & GAP

หัวใจของ skill นี้ ทำงานภายใต้ Operating Rules: **ไม่เดา input · สมมติฐานติดป้าย+โชว์ · ทุก projection = 3 scenario · แยก real/nominal** ค่าสมมติฐานตัวอย่างในไฟล์นี้เป็น **ค่าตั้งต้นที่ปรับได้ ไม่ใช่ตัวเลขทางการ** ผู้ใช้ควรยืนยัน/แทนที่ด้วยค่าของตัวเอง

> **ทำงานใน real terms (ค่าเงินวันนี้) เป็นค่าเริ่มต้น** — ใช้ "ผลตอบแทนจริง (real return)" = (1+ผลตอบแทน)/(1+เงินเฟ้อ) − 1 แล้วทุกตัวเลขอยู่ในกำลังซื้อปัจจุบัน ลดการปน nominal/real ถ้าผู้ใช้อยากเห็น nominal ค่อยแปลงตอนท้ายและติดป้ายชัด

## สูตรหลัก (แสดงสมมติฐานทุกครั้งที่ใช้)

สัญลักษณ์: `PV` เงินก้อนปัจจุบัน · `PMT` เงินออม/เดือน · `r` ผลตอบแทน**จริง net (หลัง fee/ภาษีที่เกี่ยวข้อง) ต่อปี ก่อนเกษียณ** · `rr` ผลตอบแทนจริง net ต่อปี **หลังเกษียณ** (มัก < r เพราะ de-risk) · `N` จำนวนเดือนถึงเกษียณ · `n` จำนวนปี

> **อัตราต่อเดือนใช้แบบ effective (geometric):** `i = (1 + r)^(1/12) − 1` — **ไม่ใช่** `r/12` (นั่นคือ simple/APR ทำให้ FV เพี้ยนสูงเกินจริง) ใช้ `r/12` ได้เฉพาะเมื่อระบุชัดว่า `r` เป็น nominal APR หรือเป็นค่าประมาณคร่าว
> **`r`/`rr` ต้องเป็น net** — หลังหักค่าธรรมเนียมกองทุน/ภาษีที่เกี่ยวข้อง ถ้าผู้ใช้ให้ gross ต้องเตือนว่า gross ทำให้ **ประเมิน corpus สูงเกินจริง** (ผลจริงต่ำกว่า) · **`r` (ก่อน) กับ `rr` (หลัง) ห้ามสมมติเท่ากันเอง**

1. **เงินก้อนปัจจุบันโตไปถึงวันเกษียณ**
   `FV_corpus = PV × (1 + r)^n`

2. **เงินออมรายเดือนสะสม (annuity)**
   `FV_savings = PMT × [ ((1 + i)^N − 1) / i ]`

3. **เงินก้อนที่จะมี ณ วันเกษียณ** = `FV_corpus + FV_savings`

4. **เงินก้อนเป้าหมาย (Retirement Number)** — สองวิธี เลือกตามบริบท:
   - **Withdrawal-rate method (เร็ว):** `Number = รายจ่ายต่อปีหลังเกษียณ (real) / SWR`
     เช่น SWR 4% → ต้องมี ~25× ของรายจ่ายต่อปี (SWR เป็นสมมติฐาน ไม่ใช่กฎตายตัว — ดู decumulation.md ถึงข้อจำกัด)
   - **Depletion method (ละเอียดกว่า):** หาเงินก้อนที่ถอนรายปี (real) ได้ตลอด `m` ปีหลังเกษียณ ที่ผลตอบแทนจริงหลังเกษียณ `rr`
     `Number = W × [ 1 − (1 + rr)^(−m) ] / rr`  (เมื่อ `rr > 0`; ถ้า `rr = 0` ใช้ `Number = W × m`)
     โดย `W` = รายจ่ายต่อปีหลังเกษียณ **หักรายได้หลังเกษียณอื่น** (บำนาญ/ประกันสังคม/กบข/ค่าเช่า) แล้ว

5. **เงินออมต่อเดือนที่ต้องมี — แยก total กับ additional**
   - **Total required** (สตรีมเงินออมทั้งหมดที่ต้องมี ถ้านับเฉพาะ corpus ปัจจุบันโตเอง):
     `PMT_total = max(0, (Number − FV_corpus) × i / ((1 + i)^N − 1))`
   - **Additional needed** (ส่วนที่ต้อง**เพิ่ม**จากที่ออมอยู่แล้ว `PMT_current`):
     `PMT_additional = max(0, (Number − FV_current_path) × i / ((1 + i)^N − 1))`
     โดย `FV_current_path = FV_corpus + FV_savings(PMT_current)`
   > **`max(0, …)`** กันค่าติดลบ — กรณี **surplus** (FV ≥ Number) ให้รายงานเป็น surplus (เกินเป้าเท่าไร/ออมลดได้/เกษียณเร็วได้) **ไม่แสดง PMT additional ติดลบ** · อย่าสับสน total กับ additional — ผู้ใช้ที่ออมอยู่แล้วสนใจ additional

> **หักรายได้หลังเกษียณอื่นก่อนเสมอ** — corpus ต้อง cover แค่ส่วนที่บำนาญ/ประกันสังคม/ค่าเช่า cover ไม่ถึง ลืมข้อนี้ = ประเมิน Number สูงเกินจริงมาก

### 6. Bridge period — ช่วงเกษียณก่อนรายได้เริ่ม (บังคับเมื่อเข้าข่าย)

ถ้า **อายุเกษียณ < อายุที่รายได้แต่ละแหล่งเริ่มจ่าย** (บำนาญ/ประกันสังคม/RMF ขายได้/annuity) ช่วงระหว่างนั้นคือ **bridge** ที่ corpus ต้องแบกรายจ่ายเต็ม (ยังไม่มีรายได้อื่นมาช่วย) — ไม่โมเดล = ประเมินเงินที่ต้องมีต่ำเกินจริง (critical-fail)

ดึง **income-start schedule** ก่อน: อายุเริ่มของแต่ละแหล่ง · จำนวนปี bridge `b` ของแต่ละแหล่ง · รายจ่ายที่ต้อง cover ช่วง bridge

> **หลายแหล่ง = หลาย segment ไม่ใช่ `b` เดียว:** ถ้ารายได้เริ่มคนละอายุ ให้แบ่ง bridge เป็นช่วงตามจุดที่รายได้แต่ละตัวเริ่ม แล้วคิด **net W แยกแต่ละ segment** (รายจ่าย − รายได้ที่เริ่มแล้ว ณ ช่วงนั้น) เช่น เกษียณ 52 / RMF 55 / ประกันสังคม 60 / บำนาญ 65 → segment **52–55, 55–60, 60–65** (net W ลดลงทีละขั้นเมื่อแต่ละแหล่งเริ่ม) คิดลดแต่ละ segment กลับมาที่วันเกษียณแล้วค่อยรวม

- เงินที่ bridge ต้องการ (real, **มูลค่า ณ วันเกษียณ**) ≈ ผลรวมรายจ่ายช่วง bridge **หักเฉพาะรายได้ที่เริ่มแล้ว** คิดลดด้วย `rr` (ต่อ segment ถ้ามีหลายช่วง):
  `Bridge_need = W_bridge × [ 1 − (1 + rr)^(−b) ] / rr`  (rr=0 → `W_bridge × b`)
- หลัง bridge ภาระ corpus ลดลงเพราะรายได้แหล่งนั้นเริ่มจ่าย → **คำนวณ Number เป็นช่วง ๆ** (bridge แพงกว่าต่อปี เพราะ corpus แบกคนเดียว) อย่าใช้รายจ่าย net เดียวตลอด
- **ต้องตรงฐานเวลาเดียวกัน:** ถ้าคำนวณ "เงินที่ต้องมีช่วงหลัง bridge" ที่ **อายุเริ่มรับรายได้** (post-bridge need) ต้อง **discount กลับมาที่วันเกษียณ** ก่อนนำไปรวม:
  `PostBridge_at_retirement = PostBridge_need × (1 + rr)^(−b)`
  `Number(ณ วันเกษียณ) = Bridge_need + PostBridge_at_retirement`  — อย่าบวกค่าที่อยู่คนละฐานเวลาตรง ๆ
- **Decision lever ที่ bridge เปิดให้เห็น:** เลื่อนวันเริ่มรับบำนาญ vs กันเงิน bridge แยกถัง (ดู bucket layer ใน allocation-vehicles.md) vs เลื่อนอายุเกษียณ

## NUM — คำนวณเงินก้อนเป้าหมาย

**Input ขั้นต่ำ:** รายจ่ายเป้าหมายหลังเกษียณ (real) + ปี/อายุที่เกษียณ + อายุที่วางแผนถึง
**ถ้าขาดรายจ่ายเป้าหมาย** → ทำได้แค่ illustrative ที่ติดป้าย หรือชวนผู้ใช้ประเมินจากรายจ่ายปัจจุบัน (ปรับด้วยสิ่งที่จะหาย/เพิ่มหลังเกษียณ เช่น ผ่อนบ้านหมด vs ค่ารักษาเพิ่ม)

ขั้นตอน:
1. ยืนยันรายจ่ายต่อปีหลังเกษียณ (real) → หักรายได้หลังเกษียณอื่น → ได้ `W` ที่ corpus ต้อง cover
2. เลือก method (withdrawal-rate เพื่อภาพเร็ว / depletion เพื่อความแม่น)
3. คำนวณ **3 scenario** ของ Number (ดูตารางล่าง)
4. แสดงผลเป็นช่วง + ระบุว่าตัวแปรไหนดัน Number มากสุด

## GAP — on track ไหม / ต้องเพิ่มเท่าไร

1. คำนวณ "เงินก้อนที่จะมี" จากเส้นทางปัจจุบัน (PV + PMT ปัจจุบัน) → FV
2. **เช็ก bridge:** ถ้าเกษียณก่อนรายได้เริ่ม บวก `Bridge_need` เข้าเป้าหมาย (ดูสูตร 6) — Number ที่แท้จริง = corpus หลัง bridge + เงิน bridge แยก
3. เทียบกับ Number (รวม bridge) → **surplus / shortfall** (เป็น %)
4. ถ้า shortfall → คำนวณ `PMT_additional` (ส่วนที่ต้องเพิ่มจากที่ออมอยู่) **และ** เสนอ lever อื่นที่ไม่ใช่แค่ออมเพิ่ม:
   - ออมเพิ่ม (เท่าไร)
   - เลื่อนเกษียณออก (ปีเดียวบางทีขยับ Number เยอะ เพราะออมต่อ + ถอนสั้นลง)
   - ลดรายจ่ายเป้าหมาย (ลด Number โดยตรง)
   - ผลตอบแทน/allocation (lever ที่ควบคุมได้น้อยสุด + เพิ่มความเสี่ยง — ระวังพึ่งตัวนี้)
4. แสดง trade-off ของแต่ละ lever ไม่ฟันธงว่าเลือกอันไหน → ส่งต่อ DEC

## 3-Scenario Discipline (บังคับ)

ทุก projection ออกอย่างน้อย 3 เส้น โดย **เปลี่ยนสมมติฐานที่ sensitive ที่สุด** (ในโหมด real terms คือ real return ก่อน/หลัง, อายุขัย m, withdrawal rate, savings rate — เงินเฟ้อ**ไม่ใช่ dial แยก** เพราะถูกหักอยู่ใน real return แล้ว):

| Scenario | ปรับอะไร (ตัวอย่างค่าตั้งต้น — ผู้ใช้ปรับได้) | สื่ออะไร |
|---|---|---|
| **Conservative** | real return **ก่อน** ~1–2% / **หลัง** ~0–1.5%, withdrawal 3.0–3.5%, อายุยืน (m สูง) | แผนยังรอดถ้าทุกอย่างไม่เป็นใจไหม |
| **Base** | สมมติฐานกลางที่ผู้ใช้ยอมรับได้ (ก่อน/หลัง แยกค่า) | กรณีฐานสำหรับวางแผน |
| **Optimistic** | real return **ก่อน** ~4–5% / **หลัง** ~3–4%, withdrawal 4%+, อายุตามคาด | ภาพ upside — ไม่ใช่เป้าที่ควรพึ่ง |

> **เงินเฟ้ออยู่ใน real return แล้ว — อย่านับซ้ำ:** ในโหมด real terms การลด real return ก็คือสมมติว่าเงินเฟ้อแย่/ผลตอบแทนแย่อยู่แล้ว **ห้ามปรับ "เงินเฟ้อสูงขึ้น" เป็น dial แยกในซีนาริโอเดียวกัน** (ปนกัน = นับเงินเฟ้อซ้ำ) เงินเฟ้อจะเป็น dial แยกได้เฉพาะเมื่อ (ก) ทำงานใน nominal หรือ (ข) โมเดลเงินเฟ้อเฉพาะหมวดที่โตเกิน CPI ทั่วไป (เช่นค่ารักษา) — กรณีนั้นปรับ **รายจ่าย real** ของหมวดนั้น ไม่ใช่ปรับ real return
> ค่า % ข้างบนเป็น **ตัวอย่างเชิงโครงสร้าง net** ไม่ใช่คำแนะนำว่าตลาดจะให้เท่าไร · ติดป้าย `[คาดเดา]` · **ผลตอบแทนก่อน/หลังเกษียณตั้งแยกกันเสมอ**

**Sensitivity note (บังคับ):** ปิดท้ายด้วยบรรทัดว่า "ตัวแปรไหนขยับผลมากสุด" เช่น *"shortfall ไวต่อ real return มากกว่าจำนวนเงินออม — สมมติฐานผลตอบแทนคือจุดเปราะของแผนนี้"* เพื่อให้ผู้ใช้รู้ว่าต้องระวังตรงไหน

## Output contract (NUM / GAP)
1. **Assumption Ledger** (จาก SKILL.md) — โชว์ค่าที่ใช้ + real/nominal + ผู้ใช้ยืนยันหรือยัง
2. Number หรือ surplus/shortfall เป็น **3 scenario** (ช่วง ไม่ใช่เลขเดียว) `[คำนวณ]`
3. ถ้า GAP shortfall → ตาราง lever (ออมเพิ่ม/เลื่อนเกษียณ/ลดรายจ่าย/ผลตอบแทน) + trade-off
4. Sensitivity line — ตัวแปรที่เปราะสุด
5. ข้อมูล/สมมติฐานที่ขาด + ผลต่อความเชื่อถือ
6. ความมั่นใจ (band) + ส่งต่อ ALLOC/DEC/DRAW

## Anti-patterns (NUM / GAP)
- ให้ Number เลขเดียวดูแม่น ทั้งที่ผลตอบแทน/เงินเฟ้อเป็นสมมติฐาน
- ลืมหักบำนาญ/ประกันสังคม → Number พองเกินจริง
- ปน real/nominal (คำนวณ corpus แบบ nominal แต่เทียบรายจ่าย real)
- เสนอแต่ "ออมเพิ่ม" โดยไม่โชว์ lever เลื่อนเกษียณ/ลดรายจ่าย
- ตั้งผลตอบแทน optimistic เป็น base โดยไม่บอกว่าเป็นการพึ่งโชค
- ไม่บอก sensitivity ผู้ใช้เลยไม่รู้ว่าจุดไหนเปราะ
- ใช้ `i = r/12` กับ effective annual return (FV เพี้ยนสูง) — ใช้ `(1+r)^(1/12)−1`
- ใช้ผลตอบแทน gross โดยไม่หัก fee/ภาษี (growth ดูสูงเกิน → **ประเมิน corpus สูงเกินจริง** ทำให้คิดว่าต้องออมน้อยกว่าจริง)
- สมมติ rr (หลังเกษียณ) = r (ก่อนเกษียณ)
- เกษียณก่อนรายได้เริ่มแต่ไม่โมเดล bridge → Number ต่ำเกินจริง
